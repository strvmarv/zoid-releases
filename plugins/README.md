# Contributing a plugin

This directory holds one TOML manifest per plugin, plus a generated
`index.json` that lists them all. zoid reads this catalog directly — when
someone runs `:plugin` in the app, these are the entries they browse.

## Adding a plugin

1. Write `plugins/<id>.toml` by hand (the schema is small — see below, and
   copy the shape of a manifest already in this directory).
2. Open a pull request adding just that one file.
3. Do **not** edit `plugins/index.json`. It is regenerated by CI from every
   `*.toml` here whenever one changes on `main`, and committed back
   automatically. Hand edits are overwritten.

Every manifest is reviewed before it merges. That review is the trust gate for
everyone who installs from this catalog, so the bar is: it does what it says,
and the pin is real.

## The manifest

Every manifest starts with a `[plugin]` table:

```toml
[plugin]
id          = "my-pack"        # must match the filename: plugins/my-pack.toml
schema      = 1                # always 1
kind        = ["mode"]         # "mode", "skills", or "mcp" — see below
name        = "My Pack"        # shown in the catalog list
description = "One line, shown next to the name."
license     = "MIT"            # optional, but please include it when known
```

What comes next depends on `kind`.

### `kind = ["skills"]` — a pack of skills

Add a `[source]` pointing at the upstream repo and the exact commit to install:

```toml
[source]
repo    = "owner/repo"
ref     = "a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0"   # full 40-char commit sha
subtree = "skills"                                      # optional: install only this subdirectory
```

`ref` **must be a full commit sha** — not a branch, not a tag. A branch moves,
which would silently change what people install after review. The sha is what
makes an entry reproducible and auditable.

### `kind = ["mode"]` — a pack that also installs as a mode

Same `[source]` as above, plus a `[mode]` table describing how to build the
mode. See `superpowers.toml` in this directory for a complete, working example.

### `kind = ["mcp"]` — one MCP server

An mcp manifest declares **no `[source]`** — nothing is fetched from a repo.
It describes exactly one stdio server:

```toml
[plugin]
id          = "github"
schema      = 1
kind        = ["mcp"]
name        = "GitHub"
description = "GitHub repositories, issues, and pull requests over MCP (stdio)."

[mcp.servers.github]
command = "npx"
args    = ["-y", "@modelcontextprotocol/server-github"]
env     = { GITHUB_PERSONAL_ACCESS_TOKEN = "${GITHUB_PERSONAL_ACCESS_TOKEN}" }
```

Rules for mcp entries:

- **`mcp` must be the only kind.** It does not combine with `mode` or `skills`.
  A plugin that needs two servers ships two manifests.
- **stdio only.** `command` is required and must be a real executable. HTTP/SSE
  servers are rejected.
- **Never put a secret in a manifest.** This catalog is public. Reference the
  user's own environment with `${VAR}` placeholders — zoid writes them through
  verbatim and never resolves or stores them. Before installing, it shows the
  user the exact command and flags any `${VAR}` that isn't set on their machine.

## Before you open the PR

- `id` matches the filename.
- `schema = 1`.
- `ref` is a full 40-char commit sha (skills/mode kinds).
- No secrets, no literal tokens — `${VAR}` only.
- `description` reads as one clear line; it is what people see in the catalog.
